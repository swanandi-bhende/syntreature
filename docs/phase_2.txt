1. Redesign arbitration case state machine in AIEvaluatedArbiter.sol
Add explicit lifecycle enum:
Requested, Evaluated, ExecutedRelease, ExecutedClawback, Cancelled.
Store transition timestamps to make behavior measurable:
requestedAt, evaluatedAt, executedAt.
Add execution idempotency guard so execute can only happen once.
Add case binding fields:
escrowAddress, demandId or obligationId, agent, agentId, evaluator.
Why this matters:
You need deterministic progression and one terminal action per case, otherwise judges will see it as event-only logic.

2. Add trusted escrow execution interface
Define a minimal trusted interface inside the arbiter file:
releaseFundsByArbiter(caseOrDemandId) and clawbackByArbiter(caseOrDemandId).
Add trustedEscrow mapping or immutable allowlist:
only approved escrow contracts can be targeted.
In execute function:
if shouldRelease then call escrow release function, else call clawback function.
Emit terminal event with:
caseId, escrowAddress, action, caller, proofHash, reputationWeightUsed.
Why this matters:
This is the core requirement you asked for: arbitration must actually trigger settlement.

3. Add verifiable AI proof format (signed payload + hash + metadata)
Add struct EvaluationProof:
caseId
shouldRelease
confidenceBps
model
modelVersion
sourceIdsHash
sourceCount
evidenceRoot or evidenceHash
issuedAt
expiresAt
nonce
signature
Add canonical digest function:
hashEvaluationProof(proof) returns bytes32.
Use EIP-712 style typed hash verification against oracle signer.
Enforce:
signature valid
caseId matches
not expired
nonce unused
proof hash stored onchain
Persist:
evaluationHash, sourceIdsHash, evaluator address, confidenceBps, metadata.
Why this matters:
Agent judges strongly reward verifiable machine decisions over opaque offchain claims.

4. Make reputation materially influence arbitration behavior
Keep current score storage, but use it in decision gating.
Add dynamic threshold:
requiredConfidenceBps = baseThresholdBps - reputationDiscountBps(agentScore).
Example policy:
score 800+ gets 1000 bps discount
score 500-799 gets 500 bps discount
score below 500 gets no discount and stricter sourceCount minimum.
Add minimum evidence rule:
low score agents require sourceCount >= 2 and fresher proof window.
Add execution cooldown for low score agents:
if score below risk threshold, case waits delaySeconds before execute.
Emit event with threshold used and resulting margin:
confidenceMarginBps = confidenceBps - requiredConfidenceBps.
Why this matters:
Same oracle output can produce different outcomes depending on agent reputation. That is measurable and protocol-level.

5. Separate evaluation from execution
evaluateCase:
verifies proof, computes reputation-adjusted eligibility, stores shouldRelease and verdict metadata.
executeCase:
performs escrow call and terminal transition.
Optional emergency path:
owner or guardian can cancel only in specific pre-execution states with reason code.
Why this matters:
You get cleaner auditability and easier tests for each stage.

6. Add strict invariants and custom errors
Use custom errors instead of generic require strings for audit clarity.
Core invariants:
no execute before evaluated
no evaluate after terminal state
no replayed proof nonce
escrow must be trusted
oracle signature must match configured signer
Add read helpers:
getCaseSummary(caseId)
getCaseProof(caseId)
getDecisionParams(caseId)
Why this matters:
Judges and reviewers can inspect primitive quality quickly.

Happy path release execution

Happy path clawback execution

Assert terminal state ExecutedClawback.
Signature and replay protection

Reject expired proof.

Reputation-weighted behavior is real

Submit same confidence proof for both.
High-score agent passes threshold and can execute release.
Low-score agent fails threshold or requires additional sources and cannot execute.
Assert different outcomes from identical proof confidence.

Failure outcomes update reputation distinctly
On successful release path, credit score increases by configured delta.
On clawback or invalid execution attempt, score decreases by configured delta.
Assert measurable divergence in scores and qualification status.
Idempotency and terminal safety
Double execute fails.
Evaluate after execution fails.
Non-trusted escrow address fails.

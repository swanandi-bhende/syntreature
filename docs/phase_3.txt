What “non-ambiguous” means for judges

A reviewer can see exactly which transactions are live GMX open/close actions on Arbitrum.
A reviewer can reproduce how those tx hashes were produced from your script.
A reviewer can verify that trade result triggered a credit-score write path.
README language avoids “demo-like” claims and clearly labels anything mock/test.

1) Update script architecture in deployArbitrum.ts

Split execution modes explicitly:
mode=testnet-wrapper
mode=live-gmx
Fail if mode not provided.
Print mode prominently at startup.
Add hard guardrails:
If live-gmx, require chainId === 42161 (Arbitrum One) or whichever network bond.credit expects for your submission.
If on testnet and mode is live, exit with non-zero and clear error.
Require non-zero GMX router/market addresses in live mode.
Separate output artifacts by mode:
deployments/arbitrum-deployment.testnet.json
deployments/arbitrum-deployment.live.json
Keep current arbitrum-deployment.json as a symlink/copy to the active run only if you need backward compatibility, otherwise stop using it for judge proofs.
Record replayable evidence fields from each tx receipt:

openPosition.blockNumber
openPosition.blockTimestamp

openPosition.status
same fields for closePosition
Record position-state snapshot at open and close:

size delta

close price
realized PnL
Record verification links:
explorer URL per tx hash
contract address links
chain metadata used for run
Emit machine-readable “proof manifest” block:

generatedAt
scriptVersion (git commit SHA if available)
network, chainId
evidenceComplete: true|false

5) ERC-8004 linkage pattern (mandatory for your acceptance criteria)

Choose one of these concrete models:
Direct: GMX manager calls ERC-8004 registry on close.
Indirect: GMX manager emits canonical trade event, arbiter/credit updater contract writes ERC-8004 score.
For judge clarity, ensure one transaction can be traced:
close tx -> trade finalized event -> score update tx
Include both in artifact and README with shared positionKey or tradeId.

Add deterministic score delta policy in docs:
profitable close: +X
loss close: -Y
no close / timeout: Z
This makes the update explainable and non-arbitrary.
6) Acceptance test checklist before you mark Phase 3 done

Script-level:
live-gmx mode fails on wrong chain.
script fails with zero router/market addresses.
artifact contains all required fields.
Contract-level:
open/close events include enough detail to reconstruct position lifecycle.
trade result emits linkage id used in score update path.
Evidence-level:
open and close tx hashes resolve on explorer.
timestamps recorded and readable.
score update proof exists and references trade linkage id.
README-level:
no mixed wording between live and wrapper/testnet evidence.
reviewer can replay command sequence without guessing.

7) Risks that will hurt scoring if not fixed

Keeping one mixed artifact for both testnet and live runs.
Claiming “live GMX” while using zero-address routers.
No explicit tx timestamp/block metadata.
No deterministic linkage between close result and score update.
README that blends “example flow” and “proof flow”.
